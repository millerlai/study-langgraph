## 1. LangGraph 是什麼

### 一句話定義

> LangGraph 是一個**低階的、有狀態的 AI Agent 編排框架與執行引擎（Runtime）**，專門用來建構、管理與部署**長時間運行的 stateful agent**。

### 核心理念

LangGraph **不是**一個高階的 AI 應用框架——它不會幫你抽象 prompt、不會預設 agent 架構、也不會替你決定工作流程。它提供的是最底層的**編排原語（orchestration primitives）**：

| 原語 | 說明 |
|------|------|
| **Node（節點）** | 一個 Python 函數，代表一個工作單元（呼叫 LLM、執行工具、資料處理） |
| **Edge（邊）** | 節點之間的連接，決定執行順序（固定邊 or 條件邊） |
| **State（狀態）** | 在節點之間傳遞的共享資料結構（`TypedDict` 或 Pydantic Model） |
| **Checkpoint（檢查點）** | 每個 super-step 後的 State 快照，支援持久化與恢復 |

你用這些原語像「積木」一樣自己搭建 agent 的控制流程——完全由你決定。

### LangGraph 五大核心能力

```
┌─────────────────────────────────────────────────────────┐
│                    LangGraph 核心能力                    │
├─────────────────┬───────────────────────────────────────┤
│ Durable         │ Agent 可在失敗後自動從最近的            │
│ Execution       │ checkpoint 恢復，而非重頭開始           │
├─────────────────┼───────────────────────────────────────┤
│ Human-in-the-   │ 在任意執行點暫停，讓人類檢視或           │
│ Loop            │ 修改 agent state 後再繼續              │
├─────────────────┼───────────────────────────────────────┤
│ Memory          │ 短期記憶（session 內）+                │
│                 │ 長期記憶（跨 session 持久化）           │
├─────────────────┼───────────────────────────────────────┤
│ Observability   │ 深度整合 LangSmith，可視覺化            │
│                 │ 執行路徑、State 變化、效能指標           │
├─────────────────┼───────────────────────────────────────┤
│ Production      │ 可擴展的基礎設施，專為 stateful         │
│ Deployment      │ 長時間工作流程設計                      │
└─────────────────┴───────────────────────────────────────┘
```

### 兩種 API 風格

LangGraph 提供兩種建構方式，開發者可依據需求選擇：

#### Graph API（宣告式）

透過明確宣告節點與邊來定義工作流程：

```python
from langgraph.graph import StateGraph, START, END

builder = StateGraph(MyState)
builder.add_node("node_a", func_a)
builder.add_node("node_b", func_b)
builder.add_edge(START, "node_a")
builder.add_conditional_edges("node_a", routing_func, ["node_b", END])
builder.add_edge("node_b", END)

graph = builder.compile()
```

**適用場景**：複雜的多分支、平行、子圖組合的工作流程。

#### Functional API（函數式）

用 Python 裝飾器和原生控制流（if/while）來定義邏輯：

```python
from langgraph.func import entrypoint, task

@task
def step_a(input):
    return do_something(input)

@task
def step_b(input):
    return do_another(input)

@entrypoint()
def my_workflow(input):
    a = step_a(input).result()
    if some_condition(a):
        return step_b(a).result()
    return a
```

**適用場景**：簡單線性流程，或熟悉 Python 原生控制流的開發者。

## 2. 與 LangChain 的關係與差異

### 定位差異

```
┌──────────────────────────────────────────────────┐
│              LangChain 生態系架構                   │
│                                                    │
│  ┌────────────┐  ┌────────────┐  ┌─────────────┐ │
│  │ LangChain  │  │ LangGraph  │  │  LangSmith  │ │
│  │            │  │            │  │             │ │
│  │ 高階元件    │  │ 低階編排    │  │ 可觀測性    │ │
│  │ 模型/工具   │  │ 狀態管理    │  │ 追蹤/評估   │ │
│  │ Prompt 模板 │  │ 持久化執行  │  │ 視覺化除錯  │ │
│  │ Retriever  │  │ 圖結構流程  │  │ 效能監控    │ │
│  └────────────┘  └────────────┘  └─────────────┘ │
└──────────────────────────────────────────────────┘
```

### 詳細比較

| 面向 | LangChain | LangGraph |
|------|-----------|-----------|
| **抽象層級** | 高階：提供可重用元件、chain、prompt 模板 | 低階：提供圖原語（node、edge、state） |
| **核心關注** | LLM 應用元件與整合（model、retriever、tool） | Agent 控制流、狀態管理、持久化執行 |
| **Agent 建構** | 提供預建的 agent 類型（如 `create_react_agent`） | 你自己從圖原語建構 agent 迴圈 |
| **控制程度** | 較少——框架做較多決定 | 完全控制——你決定每一步 |
| **目標使用者** | 快速原型、入門開發者 | 需要精細控制的進階開發者 |
| **是否獨立** | 獨立框架 | **完全獨立**，不依賴 LangChain |

### 關鍵觀念：LangGraph 不需要 LangChain

LangGraph 是**完全獨立**的框架。你可以：

- 直接用 OpenAI / Anthropic SDK 呼叫 LLM，不需要 LangChain 的 model wrapper
- 自己定義工具函數，不需要 `@tool` 裝飾器
- 用原生 dict 管理訊息，不需要 `HumanMessage` / `AIMessage`

但如果你同時使用 LangChain，它們整合得很好：

```python
# 搭配 LangChain 使用（非必要，但很方便）
from langchain.chat_models import init_chat_model
from langchain_core.tools import tool

model = init_chat_model("claude-sonnet-4-6")

@tool
def search(query: str) -> str:
    """搜尋網路"""
    return web_search(query)

model_with_tools = model.bind_tools([search])
```

### 與其他框架的比較

| 框架 | 特色 | 與 LangGraph 的差異 |
|------|------|---------------------|
| **CrewAI** | 角色導向的多 agent 框架，為 agent 分配「角色」（研究員、寫手等） | 更高階、更有主見；LangGraph 不預設角色概念，給你原始圖原語 |
| **AutoGen** | 微軟出品，專注多 agent 對話模式，自動回覆機制 | 偏向對話模式；LangGraph 更通用，可建模任何圖結構工作流 |
| **LangGraph** | 低階圖原語，不預設架構 | 最靈活但需要更多手動建構；你可以用它實現 CrewAI 或 AutoGen 的模式 |

**總結**：LangGraph 是「樂高積木」，CrewAI/AutoGen 是「預組裝模型」。你用 LangGraph 可以搭出任何架構，但需要自己設計。

## 3. 核心設計靈感

LangGraph 的架構源自三個成熟的分散式計算系統：

### 3.1 Google Pregel（執行模型）

[Pregel](https://research.google/pubs/pub36726/) 是 Google 於 2010 年發表的大規模圖處理系統。LangGraph 借用了它的 **message-passing 執行模型**：

#### Super-step 執行機制

```
Super-step 0          Super-step 1           Super-step 2
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│  Node A 執行  │      │  Node B 執行 │      │  Node A 執行 │
│  Node C 執行  │─────>│  Node D 執行 │─────>│    (終止)    │
│  (平行)       │      │  (平行)      │      │              │
└──────────────┘      └──────────────┘      └──────────────┘
    checkpoint             checkpoint             checkpoint
```

**核心規則：**

1. 計算以離散的 **super-step** 推進（同步迭代）
2. 節點完成後，沿著邊 **發送訊息** 給下游節點
3. 節點預設為 **`inactive`（非活躍）** 狀態
4. 收到訊息時變為 **`active`（活躍）** 並執行
5. 處理完畢後，若無外發訊息則 **投票 halt**，回到 `inactive`
6. **同一 super-step 中**的節點 → **平行執行**
7. **不同 super-step** 中的節點 → **依序執行**
8. 所有節點都 `inactive` 且無訊息在傳遞時 → **執行終止**

#### 在 LangGraph 中的體現

- `recursion_limit`（預設 1000）：限制 super-step 的最大次數，防止無限迴圈
- `RemainingSteps`：可在節點中查詢剩餘步數，實現 graceful degradation
- 平行分支中的節點自動在同一 super-step 中並行執行

### 3.2 Apache Beam（持久化與容錯）

[Apache Beam](https://beam.apache.org/) 是開源的統一資料處理管線模型。LangGraph 借用了：

| Beam 概念 | LangGraph 對應 |
|-----------|----------------|
| 資料驅動管線 | State-driven 的圖執行 |
| 管線 checkpoint | 每個 super-step 後自動 checkpoint |
| 故障恢復 | 從最近 checkpoint 自動恢復（Durable Execution） |
| 中間狀態持久化 | Checkpointer（InMemory / SQLite / PostgreSQL） |

**重點**：這就是為什麼 LangGraph 能做到「agent 跑到一半掛了，重啟後從上次斷點繼續」——這不是一般 Python 程式能做到的。

### 3.3 NetworkX（API 介面）

[NetworkX](https://networkx.org/) 是 Python 的圖分析套件。LangGraph 借用了它的 **API 命名慣例**：

```python
# NetworkX 風格
import networkx as nx
G = nx.DiGraph()
G.add_node("A")
G.add_edge("A", "B")

# LangGraph 風格（幾乎一樣）
from langgraph.graph import StateGraph
builder = StateGraph(MyState)
builder.add_node("A", func_a)
builder.add_edge("A", "B")
```

**差異**：雖然 API 長得像，但執行語義完全不同——NetworkX 做的是靜態圖分析（最短路徑、中心性等），LangGraph 做的是**訊息驅動的動態執行**。

## 4. 適用場景

### 場景一：長時間運行的 AI Agent

**問題**：傳統的 LLM 呼叫是 stateless 的一次性請求。如果你的 agent 需要執行多個步驟、可能跑好幾分鐘甚至好幾天，怎麼辦？

**LangGraph 解法**：Durable Execution + Checkpointing

```python
from langgraph.checkpoint.memory import InMemorySaver

checkpointer = InMemorySaver()
graph = builder.compile(checkpointer=checkpointer)

# 第一次呼叫——執行到一半自動存檔
config = {"configurable": {"thread_id": "session-001"}}
result = graph.invoke({"messages": [HumanMessage("開始研究")]}, config)

# 即使程式重啟，用相同 thread_id 即可從斷點繼續
result = graph.invoke({"messages": [HumanMessage("繼續上次的工作")]}, config)
```

### 場景二：多步驟工具呼叫工作流

**經典的 ReAct 模式**：LLM 思考 → 呼叫工具 → 觀察結果 → 再思考 → …直到完成

```python
from langgraph.graph import StateGraph, START, END, MessagesState

def llm_call(state: MessagesState):
    """LLM 決定下一步"""
    response = model_with_tools.invoke(state["messages"])
    return {"messages": [response]}

def tool_node(state: MessagesState):
    """執行 LLM 選擇的工具"""
    # 處理 tool calls...
    return {"messages": tool_results}

def should_continue(state: MessagesState):
    """路由：有 tool call 就繼續，否則結束"""
    last_message = state["messages"][-1]
    if last_message.tool_calls:
        return "tool_node"
    return END

# 建構圖
builder = StateGraph(MessagesState)
builder.add_node("llm_call", llm_call)
builder.add_node("tool_node", tool_node)
builder.add_edge(START, "llm_call")
builder.add_conditional_edges("llm_call", should_continue, ["tool_node", END])
builder.add_edge("tool_node", "llm_call")

agent = builder.compile()
```

**執行流程圖：**

```
START
  │
  v
┌──────────┐    有 tool_calls    ┌───────────┐
│ llm_call │ ──────────────────> │ tool_node │
│          │ <────────────────── │           │
└──────────┘                     └───────────┘
  │
  │ 無 tool_calls
  v
 END
```

### 場景三：Human-in-the-Loop 審核

**情境**：Agent 要執行敏感操作（例如刪除資料、發送郵件）前，先暫停讓人類審核，確認後才繼續。

```python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.types import interrupt, Command
from langgraph.checkpoint.memory import InMemorySaver

# 1. 定義 State
class MyState(TypedDict):
    request: str       # 使用者的請求
    result: str        # 最終結果
    approved: bool     # 是否通過審核

# 2. 定義 Nodes
def analyze_request(state: MyState) -> dict:
    """分析使用者請求，判斷要執行什麼操作"""
    request = state["request"]
    return {"result": f"準備執行：{request}"}

def sensitive_action(state: MyState) -> dict:
    """執行敏感操作前，先暫停讓人類審核"""
    # interrupt() 會暫停整個 graph，等待人類回應
    # 傳入的字串是給人類看的提示訊息
    human_decision = interrupt(
        f"[WARNING] 即將執行敏感操作：{state['result']}\n"
        f"請輸入 'approved' 核准，或其他任意值拒絕。"
    )

    if human_decision == "approved":
        return {
            "result": f"[OK] 已執行：{state['request']}",
            "approved": True,
        }
    else:
        return {
            "result": f"[CANCELLED] 已取消：{state['request']}",
            "approved": False,
        }

def report(state: MyState) -> dict:
    """產生最終報告"""
    status = "核准並執行" if state["approved"] else "已被拒絕"
    return {"result": f"最終結果 — {status}：{state['result']}"}

# 3. 建構 Graph
builder = StateGraph(MyState)
builder.add_node("analyze", analyze_request)
builder.add_node("sensitive", sensitive_action)   # 這個節點會暫停
builder.add_node("report", report)

builder.add_edge(START, "analyze")
builder.add_edge("analyze", "sensitive")
builder.add_edge("sensitive", "report")
builder.add_edge("report", END)

# 4. 編譯時必須指定 checkpointer（interrupt 需要持久化狀態）
checkpointer = InMemorySaver()
graph = builder.compile(checkpointer=checkpointer)

# 5. 執行——會在 sensitive_action 的 interrupt() 處暫停
config = {"configurable": {"thread_id": "review-001"}}
result = graph.invoke(
    {"request": "刪除所有過期使用者", "result": "", "approved": False},
    config,
)
print(result)
# 此時 graph 暫停，result 會包含到目前為止的 state

# 6. 人類審核後，用 Command(resume=...) 恢復執行
result = graph.invoke(
    Command(resume="approved"),   # 將 "approved" 傳回給 interrupt()
    config,                       # 同一個 thread_id，接續上次的執行
)
print(result)
# {'request': '刪除所有過期使用者', 'result': '最終結果 — 核准並執行：...', 'approved': True}
```

> 📄 完整範例程式碼：[1.1-example-human-in-the-loop.py](./1.1-example-human-in-the-loop.py)

**執行流程圖：**

```
START
  │
  v
┌──────────┐     ┌─────────────┐     ┌────────┐
│ analyze  │ ──> │  sensitive  │ ──> │ report │ ──> END
└──────────┘     └──────┬──────┘     └────────┘
                        │
                   interrupt()
                   ⏸️ 暫停等待
                        │
                 Command(resume=...)
                   ▶️ 恢復執行
```

**重點提醒：**
- 使用 `interrupt()` 必須在 `compile()` 時指定 `checkpointer`，因為暫停後的 state 需要被持久化保存
- 恢復時必須使用**相同的 `thread_id`**，這樣 LangGraph 才知道要接續哪次執行
- `Command(resume=value)` 中的 `value` 會成為 `interrupt()` 的回傳值

### 場景四：Multi-Agent 協作

**情境**：建立一個內容產出團隊——由 Supervisor（主管）判斷下一步該交給誰，
Research Agent 負責蒐集資料，Writer Agent 負責撰寫文章，最後由 Supervisor 決定是否完成。

```python
from typing import Annotated, TypedDict, Literal
from langgraph.graph import StateGraph, START, END
import operator

# 1. 定義共享 State
class TeamState(TypedDict):
    topic: str                                          # 要處理的主題
    research: str                                       # 研究結果
    draft: str                                          # 文章草稿
    messages: Annotated[list[str], operator.add]        # 工作日誌（用 reducer 累加）
    next_agent: str                                     # Supervisor 指派的下一位

# 2. Supervisor（主管）——決定下一步交給誰
def supervisor(state: TeamState) -> dict:
    """根據目前進度決定下一步"""
    if not state.get("research"):
        # 還沒研究 → 交給 researcher
        return {
            "next_agent": "researcher",
            "messages": ["[Supervisor] 請 Researcher 先蒐集資料"],
        }
    elif not state.get("draft"):
        # 有研究但沒草稿 → 交給 writer
        return {
            "next_agent": "writer",
            "messages": ["[Supervisor] 資料已備，請 Writer 撰寫文章"],
        }
    else:
        # 都完成了 → 結束
        return {
            "next_agent": "FINISH",
            "messages": ["[Supervisor] 文章完成，任務結束"],
        }

# 3. Research Agent
def researcher(state: TeamState) -> dict:
    """蒐集主題相關資料"""
    topic = state["topic"]
    # 實際應用中這裡會呼叫 LLM 或搜尋工具
    research = f"關於「{topic}」的研究摘要：這是一個重要的技術趨勢..."
    return {
        "research": research,
        "messages": [f"[Researcher] 已完成「{topic}」的研究"],
    }

# 4. Writer Agent
def writer(state: TeamState) -> dict:
    """根據研究結果撰寫文章"""
    research = state["research"]
    # 實際應用中這裡會呼叫 LLM 生成文章
    draft = f"文章草稿：根據研究——{research}——我們可以得出以下結論..."
    return {
        "draft": draft,
        "messages": ["[Writer] 已完成文章草稿"],
    }

# 5. 路由函數——根據 Supervisor 的決定導向對應節點
def route_next(state: TeamState) -> Literal["researcher", "writer", "__end__"]:
    next_agent = state["next_agent"]
    if next_agent == "FINISH":
        return END
    return next_agent

# 6. 建構 Graph
builder = StateGraph(TeamState)

builder.add_node("supervisor", supervisor)
builder.add_node("researcher", researcher)
builder.add_node("writer", writer)

# 起點 → Supervisor
builder.add_edge(START, "supervisor")

# Supervisor → 條件路由（交給 researcher / writer / 結束）
builder.add_conditional_edges("supervisor", route_next, ["researcher", "writer", END])

# 各 Agent 完成後 → 回到 Supervisor 重新決策
builder.add_edge("researcher", "supervisor")
builder.add_edge("writer", "supervisor")

graph = builder.compile()

# 7. 執行
result = graph.invoke({
    "topic": "LangGraph 多代理架構",
    "research": "",
    "draft": "",
    "messages": [],
    "next_agent": "",
})

print(result["draft"])
print("\n--- 工作日誌 ---")
for msg in result["messages"]:
    print(msg)
```

**完整執行順序：**

```
第 1 輪：Supervisor → "還沒研究" → 交給 Researcher
第 2 輪：Researcher → 完成研究 → 回到 Supervisor
第 3 輪：Supervisor → "有研究沒草稿" → 交給 Writer
第 4 輪：Writer → 完成文章 → 回到 Supervisor
第 5 輪：Supervisor → "都完成了" → END
```

**重點提醒：**
- Supervisor 是整個系統的**決策中心**，每一輪都回到它來判斷下一步
- `add_conditional_edges` 讓 Supervisor 根據 `next_agent` 動態選擇路由
- `messages` 欄位使用 `Annotated[list[str], operator.add]` 作為 reducer，每個節點的回傳會**累加**而非覆蓋
- 實際應用中，Supervisor 通常會呼叫 LLM 來做決策，而非寫死的 if/else

### 場景五：動態平行處理（Map-Reduce）

**情境**：使用者提交多篇文章，系統需要**平行**摘要每篇文章，最後**彙整**成一份總結報告。
文章數量在定義 Graph 時未知，需要在執行時動態決定。

```python
from typing import Annotated, TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send
import operator

# 1. 定義 State
class OverallState(TypedDict):
    """主 Graph 的 State"""
    articles: list[str]                                     # 輸入：多篇文章
    summaries: Annotated[list[str], operator.add]           # 各篇摘要（reducer 累加）
    final_report: str                                       # 最終彙整報告

class ArticleState(TypedDict):
    """單篇文章處理用的 State（每個平行分支獨立）"""
    article: str                                            # 單篇文章內容

# 2. 收集節點——動態 fan-out，為每篇文章建立一個平行分支
def collect_articles(state: OverallState):
    """用 Send 為每篇文章動態派發一個 summarize 節點"""
    return [
        Send("summarize", {"article": article})
        for article in state["articles"]
    ]

# 3. 摘要節點——每個平行分支獨立執行
def summarize(state: ArticleState) -> dict:
    """摘要單篇文章（實際應用會呼叫 LLM）"""
    article = state["article"]
    summary = f"摘要：{article[:20]}..."  # 模擬 LLM 摘要
    # 回傳到主 State 的 summaries 欄位（透過 reducer 累加）
    return {"summaries": [summary]}

# 4. 彙整節點——所有平行分支完成後執行
def aggregate(state: OverallState) -> dict:
    """將所有摘要彙整成最終報告"""
    all_summaries = "\n".join(
        f"  {i+1}. {s}" for i, s in enumerate(state["summaries"])
    )
    report = f"總結報告（共 {len(state['summaries'])} 篇）:\n{all_summaries}"
    return {"final_report": report}

# 5. 建構 Graph
builder = StateGraph(OverallState)

builder.add_node("summarize", summarize)
builder.add_node("aggregate", aggregate)

# START → 動態 fan-out（用 conditional_edges + Send）
builder.add_conditional_edges(START, collect_articles, ["summarize"])

# 所有 summarize 分支完成後 → aggregate
builder.add_edge("summarize", "aggregate")

# aggregate → END
builder.add_edge("aggregate", END)

graph = builder.compile()

# 6. 執行
result = graph.invoke({
    "articles": [
        "LangGraph 是一個低階的 AI agent 編排框架，專門用來建構 stateful agent...",
        "LangChain 提供高階元件，包括 model wrapper、prompt template、retriever...",
        "LangSmith 是一個可觀測性平台，提供 tracing、evaluation、除錯功能...",
    ],
    "summaries": [],
    "final_report": "",
})

print(result["final_report"])
# 總結報告（共 3 篇）:
#   1. 摘要：LangGraph 是一個低階的 AI...
#   2. 摘要：LangChain 提供高階元件，包括...
#   3. 摘要：LangSmith 是一個可觀測性平台...
```

> 📄 完整範例程式碼：[1.1-example-map-reduce.py](./1.1-example-map-reduce.py)

**執行流程圖：**

```
START
  │
  │  collect_articles() 動態產生 Send(...)
  │
  ├─── Send("summarize", {article: 文章1}) ───┐
  │                                            │
  ├─── Send("summarize", {article: 文章2}) ───┤  同一 super-step
  │                                            │  平行執行
  ├─── Send("summarize", {article: 文章3}) ───┤
  │                                            │
  :    （文章數量在執行時才決定）                  :
  │                                            │
  │    ┌──────────┐  ┌──────────┐  ┌──────────┐│
  │    │summarize │  │summarize │  │summarize ││
  │    │  文章 1   │  │  文章 2   │  │  文章 3   ││
  │    └────┬─────┘  └────┬─────┘  └────┬─────┘│
  │         │             │             │      │
  │         └─────────────┴─────────────┘      │
  │                       │                    │
  │                       │  所有分支的 summaries
  │                       │  透過 reducer 累加
  │                       v
  │              ┌─────────────┐
  │              │  aggregate  │
  │              │  彙整報告    │
  │              └──────┬──────┘
  │                     │
  v                     v
                       END
```

**Map-Reduce 運作原理：**

```
 Map 階段（平行展開）              Reduce 階段（收斂彙整）
┌─────────────────────┐          ┌─────────────────────┐
│                     │          │                     │
│  文章1 → summarize ─┤          │                     │
│                     │ reducer  │                     │
│  文章2 → summarize ─┼─(累加)──>│     aggregate       │
│                     │          │     彙整所有摘要     │
│  文章3 → summarize ─┤          │                     │
│                     │          │                     │
└─────────────────────┘          └─────────────────────┘
  每篇獨立處理                      合併為最終報告
```

**重點提醒：**
- `Send("node_name", state_dict)` 會為每個項目建立**獨立的執行分支**，分支數量在執行時動態決定
- 每個分支使用獨立的 `ArticleState`，彼此不互相干擾
- 分支結果透過 **reducer**（`Annotated[list[str], operator.add]`）自動累加回主 State
- 所有平行分支在**同一個 super-step** 中執行，完成後才進入下一個節點（`aggregate`）
- 這比起寫 `asyncio.gather()` 的好處是：每個分支都有 checkpoint，失敗時只重試失敗的分支

### 最小完整範例

```python
from typing import Annotated, TypedDict
from langgraph.graph import StateGraph, START, END

# 1. 定義 State
class MyState(TypedDict):
    counter: int
    message: str

# 2. 定義 Node
def increment(state: MyState) -> dict:
    return {"counter": state["counter"] + 1, "message": "已加一"}

def double(state: MyState) -> dict:
    return {"counter": state["counter"] * 2, "message": "已加倍"}

# 3. 建構 Graph
builder = StateGraph(MyState)
builder.add_node("increment", increment)
builder.add_node("double", double)
builder.add_edge(START, "increment")
builder.add_edge("increment", "double")
builder.add_edge("double", END)

graph = builder.compile()

# 4. 執行
result = graph.invoke({"counter": 3, "message": ""})
print(result)
# {'counter': 8, 'message': '已加倍'}
# 流程：3 → increment(+1) → 4 → double(×2) → 8
```

## 重點摘要

| 問題 | 答案 |
|------|------|
| LangGraph 是什麼？ | 低階 stateful AI agent 編排框架 |
| 需要 LangChain 嗎？ | 不需要，完全獨立 |
| 設計靈感來源？ | Pregel（執行模型）、Beam（容錯）、NetworkX（API） |
| 什麼時候該用？ | 需要 stateful、長時間運行、人機協作、多 agent 的場景 |
| 兩種 API？ | Graph API（宣告式）和 Functional API（函數式） |
| 與 CrewAI/AutoGen 差異？ | LangGraph 更低階、更靈活，但需要更多手動建構 |
